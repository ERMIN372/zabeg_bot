# Telegram-бот «Свои в ритме города»

Бот бегового сообщества сети кулинарных лавок «Братья Караваевы»: запись на
пробежки с выбором локации, лист ожидания с таймером подтверждения,
напоминания, поддержка через админ-группу и встроенная админ-панель.

## Стек

| Компонент | Выбор |
|---|---|
| Бот | Python 3.12 + aiogram 3 (long polling) |
| БД | SQLAlchemy 2 (async) + Alembic; SQLite для разработки, PostgreSQL для прода — переключение переменной `DATABASE_URL` |
| Планировщик | отдельный процесс (`scheduler/`), состояние только в БД — рестарты ничего не теряют |
| Деплой | Docker Compose (бот + планировщик + PostgreSQL) на любом VPS |

## Структура

```
bot/            хендлеры сценариев, клавиатуры, FSM, middleware
bot/handlers/admin/   админ-панель (/admin)
db/             модели SQLAlchemy, фабрика сессий
services/       бизнес-логика: атомарная запись, лист ожидания, экспорт, тексты
scheduler/      процесс напоминаний и таймера листа ожидания
migrations/     миграции Alembic
tests/          тесты критичной логики
```

## Быстрый старт (Docker, одной командой)

1. Скопируйте `.env.example` в `.env` и заполните минимум `BOT_TOKEN`,
   `ADMIN_IDS`, `SUPPORT_CHAT_ID`, `PRIVACY_POLICY_URL`, `POSTGRES_PASSWORD`.
2. Запустите:

```bash
docker compose up -d --build
```

Compose поднимет PostgreSQL, применит миграции и запустит бота с планировщиком.
Логи: `docker compose logs -f bot scheduler`.

## Локальный запуск без Docker (SQLite)

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements-dev.txt
cp .env.example .env          # заполните BOT_TOKEN и остальное
export $(grep -v '^#' .env | xargs)   # или любой другой способ загрузки .env
alembic upgrade head          # создаст zabeg.db
python -m bot.main            # терминал 1 — бот
python -m scheduler.main      # терминал 2 — планировщик
```

## Переменные окружения

| Переменная | Назначение |
|---|---|
| `BOT_TOKEN` | токен бота от @BotFather (обязательно) |
| `DATABASE_URL` | `sqlite+aiosqlite:///./zabeg.db` или `postgresql+asyncpg://user:pass@host/db` |
| `ADMIN_IDS` | Telegram user_id админов через запятую — доступ к `/admin` |
| `SUPPORT_CHAT_ID` | ID группы, куда пересылаются вопросы поддержки |
| `PRIVACY_POLICY_URL` | ссылка на полный текст политики конфиденциальности |
| `CONSENT_VERSION` | версия текста согласия на ПДн; поднимайте при изменении политики |
| `PHOTO_MODE` | `single_link` — одна ссылка на архив; `per_event` — альбомы по мероприятиям |
| `PHOTOS_ARCHIVE_URL` | ссылка на общий архив (для `single_link` и как fallback) |
| `WAITLIST_CONFIRM_HOURS` | часов на подтверждение места из листа ожидания (по умолчанию 2) |
| `SCHEDULER_TICK_SECONDS` | период тика планировщика (по умолчанию 30) |
| `DEFAULT_TIMEZONE` | часовой пояс по умолчанию для новых мероприятий |

## Настройка поддержки

1. Создайте закрытую группу, добавьте туда бота.
2. Узнайте ID группы (например, добавив @getidsbot) и пропишите в `SUPPORT_CHAT_ID`.
3. Вопросы пользователей будут приходить в группу; ответ **reply-ем** на
   сообщение бота доставляется пользователю в личку.

## Админ-панель

Команда `/admin` (только для `ADMIN_IDS`):

* создание мероприятий: пробежка с несколькими локациями или закрытое
  мероприятие с одной локацией (дегустация и т.п.);
* редактирование названия, описания, даты/времени, часового пояса, картинки,
  ссылки на альбом;
* локации: добавление, адрес, **лимиты мест** (при увеличении лимита места
  автоматически раздаются листу ожидания), часовой пояс;
* перенос (участники получают уведомление, напоминания пересчитываются)
  и отмена мероприятия (с рассылкой);
* экспорт регистраций в CSV (открывается в Excel) — по мероприятию или все;
* редактирование текстов разделов (приветствие, «О сообществе», текст
  согласия, фото/видео, приветствие поддержки) без участия разработчика;
* пересчёт счётчиков мест (страховка).

## Mini app (веб-админка)

Помимо панели `/admin` на кнопках, есть **Telegram Mini App** — веб-интерфейс
всей админки (мероприятия, локации и лимиты, тексты разделов, экспорт,
статистика). Удобно передавать заказчику: открывается прямо в Telegram.

**Что важно про доступ:** кнопку «🚀 Открыть панель» бот показывает только в
ответе на `/admin`, а `/admin` доступен лишь `ADMIN_IDS` — обычные
пользователи её не видят. Плюс сервер на **каждый** запрос проверяет подпись
Telegram `initData` и членство в `ADMIN_IDS` (`webapp/auth.py`), поэтому даже
прямой заход по URL посторонним ничего не даёт (401).

Всё бесплатно: используется тот же VPS, TLS — Let's Encrypt через Caddy,
домен можно взять бесплатно.

### Запуск

1. **Бесплатный домен.** Зарегистрируйте поддомен на
   [DuckDNS](https://www.duckdns.org) (например `zabeg-admin.duckdns.org`) и
   укажите в нём IP этого сервера. Без регистрации — вариант
   `<IP-с-дефисами>.sslip.io`.
2. В `.env` задайте:
   ```
   WEBAPP_DOMAIN=zabeg-admin.duckdns.org
   WEBAPP_URL=https://zabeg-admin.duckdns.org
   ```
3. Откройте порты 80 и 443 на сервере (нужны Caddy для выпуска сертификата).
4. Поднимите всё как обычно:
   ```bash
   docker compose up -d --build
   ```
   Caddy сам выпустит и будет продлевать бесплатный TLS-сертификат.
5. В боте отправьте `/admin` → кнопка «🚀 Открыть панель».

Если `WEBAPP_DOMAIN`/`WEBAPP_URL` не заданы — mini app просто не участвует,
бот работает как раньше (панель `/admin` на кнопках остаётся).

> Картинку мероприятия и картинку раздела «О сообществе» по-прежнему задавайте
> в боте фото-сообщением (Telegram отдаёт картинки только как `file_id`).

## Как устроена защита от гонок

Захват места — условный атомарный `UPDATE locations SET taken = taken + 1
WHERE id = :id AND taken < capacity` в одной транзакции с созданием записи.
Из двух одновременных подтверждений на последнее место проходит ровно одно —
одинаково в SQLite и PostgreSQL, без ветвления по диалектам. Дубликаты
блокирует частичный уникальный индекс «одна активная запись на пользователя
в рамках мероприятия», поэтому двойное нажатие «Подтвердить запись»
безопасно.

Лист ожидания: активных предложений никогда не больше, чем свободных мест;
по истечении `WAITLIST_CONFIRM_HOURS` планировщик передаёт предложение
следующему в очереди, а опоздавший перемещается в конец.

## Тесты

```bash
pip install -r requirements-dev.txt
pytest
```

Покрыто: гонка за последнее место, идемпотентность двойного подтверждения,
частичный уникальный индекс, цепочка листа ожидания (предложение → истечение →
переход к следующему → подтверждение), смена локации с сохранением текущей
записи при неудаче.

## Миграции

```bash
alembic upgrade head                        # применить
alembic revision --autogenerate -m "..."    # создать новую после правки моделей
```
